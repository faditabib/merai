# Merai тАФ Architectural Decisions

## 2026-07-16 тАФ Build 6C.4: a creator type is a pointer, not an entity; the wizard is a guided writer
The onboarding wizard ships with zero new persistence surfaces. Three decisions. **(1) A creator type is a POINTER** тАФ `CREATOR_TYPES` (code catalog, 6 personas) maps each type to an existing Creator Style id + an `ai_preferences.intent` seed; `creatorTypeDefaults` resolves the bundle. No table, no migration, no new export/render surface тАФ the no-real-names CI guard extends to type ids. **(2) The wizard is a guided writer over channels that already exist:** Finish performs `creatorStyleBrandKitPatch` тЖТ `brand_kits` upsert (identity preserved by the tested resolver, wizard's name/colors/logo on top), `user_metadata` flags (`creator_type`, `creator_style`, `onboarding_completed_at`, `logo_overlay` when the style carries one), and an `ai_preferences.intent` upsert тАФ the user's explicit choice, never profiling. Skip writes ONLY the completion flag, so the wizard never re-nags and never touches data. **(3) Colors are creator-owned:** the palette prefills from the type's style but one manual edit pins it across style switches (`colorsEdited`) тАФ a style restyles the look, the creator's explicit color choices win. **Revisit when:** the wizard should seed `projects.default_aspect_ratio` (with Auto Canvas), or a queryable `profiles.creator_type` column is needed for analytics (today it rides user_metadata).

## 2026-07-12 тАФ Build 6C.3: the logo is one more full-frame PNG; overlays are additive; zero migrations
Overlay Studio adds the creator's logo/watermark and a richer lower third without redesigning the render architecture. Four decisions. **(1) The logo composites as another full-frame transparent PNG overlaid at 0:0**, exactly like the gradient and lower third тАФ the worker bakes corner-placement, size (widthPct of frame width, aspect preserved) and opacity into the PNG, so `buildExportPlan` just adds one more `overlay=0:0` and both render engines are untouched. Canonical z-order is now `video тЖТ gradient тЖТ lower third тЖТ captions тЖТ logo` (captions are the highest readability layer; logo on top for branding). **(2) Resolve the logo BEFORE planning, and skip it on ANY failure.** `render-export` downloads + rasterizes the logo before `buildExportPlan`; if the image is missing, undecodable (SVG тАФ `@napi-rs/canvas` can't rasterize it), or anything throws, the layer is dropped from both the brand config and the plan with a warning тАФ a cosmetic overlay must never fail a render. **(3) Shared `logoBox` geometry in core** (margin % of the frame, box sized by frame width) is used by the worker (canvas pixels) and mirrored by the Overlay Studio preview (CSS %), so preview = export by construction (the no-fake-preview mandate). **(4) Zero migrations:** `exports.brand.logo` and the lower-third `position`/`shape` are additive jsonb; the logo placement *default* rides `user_metadata.logo_overlay` (the 6C creator-pref pattern); the logo image already lives in `brand_kits.logo_path`. Absent logo/fields = pre-6C.3 render (regression-tested). **Revisit when:** SVG logo rasterization is needed (a rasterization step), per-project overlay overrides land, or logo placement wants a queryable `brand_kits.logo_default` column instead of user_metadata.

## 2026-07-12 тАФ Build 6C.2: Creator Styles are bundles that write existing kit fields
Creator Styles turn editing patterns into one-tap creative identities without any new runtime object, table, migration, or render code. Four decisions. **(1) A style is a curated BUNDLE, not a new entity.** `CREATOR_STYLES` (code catalog in core, 6 entries) maps each style to a real `CaptionStyleSpec` + colors + gradient + lower-third colors + recommended format; applying writes those into the existing `brand_kits` fields, so the 6B.1/6B.3 editorтЖТexport pipeline carries the look with ZERO export/worker/DB change. `creatorStyleBrandKitPatch` is a pure, tested resolver. **(2) Apply overwrites the look, preserves identity.** Colors + caption default + overlay + lower-third colors restyle; the lower-third name/title/subtitle and the logo are kept тАФ so a doctor's "Dr. Ahmad / Cardiologist" survives a restyle. In the UI, Apply seeds the Brand Kit form's live state (instant transformation) and Save commits тАФ no silent DB write, no refresh/state mismatch. **(3) No hidden profiles.** Applying is explicit; the chosen style id lives in `user_metadata` as the creator's own visible choice (a "Your style" chip), never an accumulated behavioral profile. **(4) Generic product names only**, enforced by a CI test that rejects real creator names in style ids тАФ the PRD house rule (real creators are internal visual references, never shipped). **Revisit when:** the onboarding wizard (6C.4) applies a style directly to the DB (it reuses `creatorStyleBrandKitPatch`), or a saved custom-style library is ever built (then a table).

## 2026-07-12 тАФ Build 6C.1: Tajawal is UI-only; thumbnails are client-side; zero migrations
First 6C sub-build (creator experience layer). Three decisions. **(1) The UI font (Tajawal) is distinct from the caption font (IBM Plex Sans Arabic).** The UI migrates to Tajawal via `--font-sans`/`body`, but Plex stays loaded and is pinned as a `--font-caption` CSS var that `captionSpanStyle` resolves тАФ so the live caption preview keeps using the exact family the worker rasterizes (vendored Plex) and can't silently drift from the export. Vendoring Tajawal into the caption *renderer* is deliberately deferred (new TTFs + Arabic-shaping verification). **(2) Thumbnails are client-side and disposable.** `ProjectThumbnail` grabs a first frame via a lazy `IntersectionObserver` тЖТ signed raw URL тЖТ `<video>`тЖТ`<canvas>` тЖТ cached JPEG data URL, with a branded placeholder fallback (a missing thumbnail is never an error). No `thumbnails` bucket, no worker poster job, no migration тАФ worker-generated posters remain the deferred optimization. **(3) The creator dashboard reads only existing tables.** The brand-setup nudge and thumbnails needed no new columns тАФ two extra bounded reads (brand_kit + latest uploads) resolved alongside projects and deduped in JS (no N+1). **6C.1 ships with zero migrations and no worker/render change.** **Revisit when:** thumbnail decode cost shows up at scale (then a worker poster step + bucket), or a queryable creator_type is needed (then a profiles column instead of user_metadata).

## 2026-07-12 тАФ Build 6B.3: Caption Studio is UX + one column; the worker is untouched
Caption Studio UX (creator-facing preset gallery + live controls + default preference + export style card) ships without touching the EDL, the AI Brain, the render architecture, or the worker. Five decisions. **(1) UX-only + one nullable column.** The worker already resolves `caption_config ?? token` and honors outline/scale/brand color (6B.2), so 6B.3 adds no worker code тАФ only web components, an 8-preset core catalog (`CAPTION_PRESETS`), and `brand_kits.caption_default_config jsonb` (nullable). Null default + null caption_config = byte-identical to pre-6B.3. **(2) One shared preview renderer.** `CaptionPreview`/`captionSpanStyle` mirror the server rasterizer's fields and are reused by the gallery, live preview, and editor overlay тАФ preview=export is structural, not coincidental. **(3) The default is a preference, not a library.** Exactly one `caption_default_config` per creator ("set as default"); no named-preset table (deferred). **(4) Animation is a label, not motion.** The pipeline stays one static PNG per line; the animation chip communicates feel, and animated/karaoke export stays deferred (needs render architecture тАФ gap G1). **(5) `CaptionStyleSpec.token` widened to `string`** so persisted/snapshotted specs (`exports.caption_config`, `caption_default_config`) are structurally assignable, while the built-in `CAPTION_STYLE_SPECS` keys stay strongly typed. The 8 new preset ids are additive members of the resolvable token map; the pre-6B.3 developer tokens remain resolvable for old rows but are hidden from the gallery. **Revisit when:** a saved custom-preset library is built (a `caption_presets` table reusing the same snapshot channel), or the render plan gains intra-line caption timing (then animation becomes real, not a label).

## 2026-07-12 тАФ Build 6B.2: caption presets are brand-aware via an export snapshot
Caption Studio makes captions a brand-aware preset system without touching the EDL, AI Brain, or render architecture. Four decisions. **(1) The resolved caption spec is snapshotted to `exports.caption_config` (nullable jsonb), never an EDL field.** The EDL keeps `captionStyle` as a token string; brand-colored presets can't be a token, so `captionConfigForExport(token, brandColors)` resolves them in the app (where the colors live) and stores the concrete spec on the export row. Null = the token render path, byte-identical to pre-6B.2 (regression-tested in core AND worker). Same rule as `exports.brand` (6B.1): resolve at export, so a later Brand Kit edit can't change a render that already happened. **(2) The renderer stays dumb.** The rasterizer never reads `useBrandColor`; it draws concrete colors from either the snapshot or the token spec. Brand resolution is a pure, tested core function, validated at the render boundary by `captionStyleSpecSchema` (malformed `caption_config` тЖТ PermanentJobError, `fontScale` clamped). **(3) Dropped the `brand_kits.caption_style_default` CHECK** тАФ enumerating tokens in a DB constraint forces a migration per new preset for a cosmetic column; the app validates with a zod enum that `.catch()`es unknown tokens to the default. **(4) Karaoke stays preview-only.** True per-word caption timing in export needs time-sliced PNGs = render architecture = out of scope; karaoke exports as an honest static line (documented gap G1). **Revisit when:** saved custom presets land (6B.3 тАФ a `caption_presets` table + the same `caption_config` channel), or the render plan gains intra-line caption timing (then real karaoke export).

## 2026-07-11 тАФ Build 6B.1: branding is an export-row snapshot, not an EDL field
Creator Identity Layer (Brand Kit + caption presets + brand overlays). Four binding decisions. **(1) Branding lives on `exports.brand` (nullable jsonb), snapshotted at export time тАФ never in the EDL.** `downgradeEdlV2ToV1` refuses `has-effects`, so brand overlays carried as EDL v2 effects would permanent-fail every branded export until the renderer learns tracks; and a snapshot means a later Brand Kit edit can't alter a render that already happened тАФ the exact semantics `aspect_ratio`/`caption_style` already have. Null brand = the pre-6B.1 plan, byte-identical (regression-tested in core AND worker). **(2) Brand layers ride the caption-image channel.** `buildExportPlan` (pure, unit-tested) composites in binding order `video тЖТ gradient тЖТ captions тЖТ lower third` via dynamic ffmpeg input indices; the gradient is a readability layer UNDER captions, the lower third sits ON TOP. Both render engines stage `captionImages` generically, so appending brand PNGs changed **zero** engine code. **(3) One Brand Kit per creator** (`brand_kits`, `unique(owner_id)`, owner-only RLS, upsert-on-conflict) тАФ right-sized; multi-brand orgs later. Logo is stored + previewed but NOT yet composited into the frame (lower third renders text + accent bar in this foundation). **(4) Malformed `exports.brand` is a PermanentJobError**, never a silent unbranded render тАФ the creator's branding is not something to quietly drop. **Revisit when:** the renderer learns EDL v2 tracks (overlays could move into the timeline), logo compositing/animation lands, or AI-suggested presets need per-content selection.

## 2026-07-11 тАФ Brain annotations ride inline then split; preferences are explicit-only
Build 5.6 decisions: **(1) Presentation is best-effort, mutation is strict.** The Brain annotates each command inline (title/reason/benefit/category); `parseAnnotatedPlan` splits pure commands from presentation steps тАФ a malformed annotation degrades to an empty step, never sinks a valid command; steps travel with their command through normalization. The UI computes every NUMBER (durations, counts) from the EDL; the model contributes only qualitative text тАФ invented metrics are structurally impossible to display. **(2) Feedback lives on ai_suggestions** (1:1, existing owner RLS); **preferences store only the user's explicit choice** тАФ auto mode derives a hint per-request from APPLIED suggestion goals and persists nothing, so no hidden profile can accumulate. **(3) Normalization absorbs the "already satisfied" class**: after two live findings (words in 5.5, segments in 5.6), any intent the edit already satisfies is deduplicated, while ids that exist nowhere still reject the whole plan. **Revisit when:** feedback volume justifies prompt tuning, or per-step accept/reject lands.

## 2026-07-11 тАФ AI Editing Brain: suggestion rows only, refuse-or-normalize, human applies
Build 5.5 wires user intent to the Build 5 command system: a worker job (the reserved `generate_edl` slot тАФ the AI key lives only in the worker) calls Haiku ONCE (forced tool-use, temp 0) and writes a plan into `ai_suggestions` (migration 7). **Three binding rules: (1) the Brain has no code path to EDLs** тАФ it fills suggestion rows; the EDL changes only in the editor when the owner clicks Apply (one undoable batch through the dispatcher). **(2) Refuse-or-normalize validation**: every plan passes a command-type allowlist (5 safe commands; trim/split/reorder excluded until range guardrails exist), referential checks, and a full dry-run тАФ ids that exist nowhere reject the WHOLE plan (hallucination), while intents the edit already satisfies (removing an already-removed word тАФ hit live on run 1) are deduplicated, and the stored commands are exactly the dry-run's normalized batch. **(3) Suggestions pin the saved EDL version** (auto-save before request); Apply is disabled when the edit moves past the pin тАФ no plan rebasing in v1. **Revisit when:** range-guarded trim/split lands, or the multi-track editor makes the Brain reason over EDL v2 directly.

## 2026-07-11 тАФ Build 6A: trust UI shows only real data; UX flags live in auth metadata
Three decisions from the creator-experience layer. **(1) No fake confidence:** `RemovedSegment.confidence` exists as an optional schema hook and the AI decision card renders it only when present тАФ engines start emitting it in a later build. Inventing a score would poison exactly the trust the feature builds. **(2) One decision card, every surface:** the transcript popover and timeline ghosts render the same `AiDecisionCard`, so the AI explains cuts identically everywhere. **(3) Per-user UX flags go in `auth.updateUser` user_metadata** (`onboarding_dismissed_at`), not a profiles column тАФ cross-device, zero migration, zero extra queries. Also binding: the shortcut-help gesture listens for both `?` and `╪Я` (Arabic keyboard layouts). **Revisit when:** UX flags multiply (then a preferences column/table) or engines emit confidence (then remove the "not yet emitted" caveats).

## 2026-07-11 тАФ EDL v2 (multi-track) foundation: readers-first, refusal over flattening
Build 5 introduces the multi-track EDL (explicit `timelineInMs` placement, video/audio/caption tracks, locked A/V pairs via `linkedClipId` тАФ a J/L-cut is a linked pair whose windows differ тАФ multiple assets, open effect/transition metadata) in `@merai/core/edl-v2.ts`, plus a serializable `EditCommand` surface (`edit-commands.ts`) that the editor UI and future AI re-editing share. Three binding rules: **(1) expand/contract, readers first** тАФ every jsonb reader now goes through `parseEdl`/`edlV1ViewOf` (the blind `as EdlV1` casts are gone); writers keep producing v1 until the multi-track UI build, so production data is unchanged and no DB migration exists (the JSON `version` literal from Phase 0 is the discriminator). **(2) Downgrade refuses, never flattens** тАФ `downgradeEdlV2ToV1` returns a typed reason (12 enumerated) for anything not v1-representable; the render handler turns that into a `PermanentJobError`. Silently dropping a track would corrupt a user's edit. Round-trip law `downgrade(upgrade(v1)) тЙб v1` is a test. **(3) AI edits are commands, not EDL patches** тАФ models emit `EditCommand` JSON, zod-validated, routed through the same tested ops as the UI. **Revisit when:** the multi-track UI lands (editor state moves to v2, save writes v2, planner learns tracks).

## 2026-07-10 тАФ Phase A hardening: permanent errors, over-cap download fallback, alerting, first deploys
Four decisions from the pre-first-users hardening pass:

**1. Deterministic job failures skip retries.** `PermanentJobError` (worker) short-circuits the queue: the runner hard-fails the job on attempt 1 (`failJobPermanently`), surfaces it to the exports/projects row, and fires an alert. Classified so far: missing exports/EDL/upload rows and storage size-cap rejections. **Why:** a deterministic failure re-rendered a 10-minute video 3├Ч before failing (~3 min of wasted CPU per failure). **Revisit when:** a new failure class shows up mislabeled as permanent.

**2. Over-cap exports fall back to part-split storage, reassembled in the browser.** When the storage per-file cap rejects an output, the worker stores `{export}.mp4.partN` objects (45MB each, under the free-tier 50MB cap) and records `exports.parts` (migration 6); the panel downloads all parts and hands the user ONE assembled file via a Blob. **Why:** the render succeeded тАФ the user should get their video, not an error, even before the Supabase Pro upgrade. Sub-cap exports are byte-identical to before (parts=1). **Revisit when:** Supabase Pro raises the cap тАФ the fallback goes dormant but stays as a safety net.

**3. Ops alerting is a generic webhook.** `ALERT_WEBHOOK_URL` (worker) receives permanent-failure and db-pool-error alerts with a payload carrying both Slack (`text`) and Discord (`content`) fields; log-only when unset; never throws. No email infra, no vendor SDK.

**4. Vercel deploy findings (config-as-code where possible).** Project `merai-web`, root directory `apps/web` (set via `vercel api` тАФ the CLI has no flag for it), framework pinned in `apps/web/vercel.json`; worker deploy config in `railway.json`. Two gotchas hit live: (a) `merai-web.vercel.app` is ANOTHER user's project тАФ our real domain is `merai-web-faditabibs-projects.vercel.app` until merai.studio is attached; (b) SSG locale routes break @vercel/next's routeтЖТlambda mapping ("Unable to find lambda for route: /ar/login"), so the `[locale]` layout is `force-dynamic` for now тАФ trivial pages, no meaningful cost. **Revisit when:** the builder handles Next 16 SSG locale routes, or the custom domain lands.

## 2026-07-10 тАФ Export rendering moved fully server-side; ffmpeg.wasm removed
Strategic pivot (owner decision): unit economics showed ~$1.50/subscriber/month between browser and server rendering тАФ not worth the wasm complexity (segment-wise workarounds, wasm FS management, device variance). ffmpeg.wasm is fully deleted; exports are now `render_export` jobs on the existing Railway worker, and the export panel is request + poll (same pattern as transcription) with server-checkpointed cancel. The segment-wise planner moved unchanged to @merai/core (it was already pure).

**Provider choice (owner asked: Rendi vs Very Good FFmpeg, pick by TS SDK support).** Verified from public pages 2026-07-10: Rendi has NO official SDK, ~$0.15/GB, and job-runtime caps of 1 min (free) / 10 min (Pro) тАФ genuinely risky for 10-minute renders. Very Good FFmpeg has an official TS SDK (`@verygoodffmpeg/sdk`), raw-ffmpeg commands with `{{file}}` templating, 6-hour job runtime, $0.50/GBтЖТ$0.10тЖТ$0.08 tiers, 2GB free. **VGF wins on the stated criterion and on runtime limits.** Sources: rendi.dev/pricing, verygoodffmpeg.com, renderio.dev/blogs/ffmpeg-api-pricing-compared.
**Deviation, flagged:** the DEFAULT engine is `LocalFfmpegEngine` тАФ native ffmpeg on the worker itself (one `apk add ffmpeg` in the Dockerfile). Zero marginal cost, media never leaves our storageтЖФworker path, and it was testable TODAY (a managed provider needs an account/key only the owner can create). The VGF engine is fully wired against their documented REST API (bearer + POST /api/ffmpeg + GET /api/jobs/{id}; the SDK README wasn't reachable, so no unverified SDK shapes) and activates via `VERYGOODFFMPEG_API_KEY` with zero code changes тАФ the exact house pattern (AssemblyAIтЖФmock, HaikuтЖФheuristic). Like AssemblyAI in Phase 1, the VGF path is UNVERIFIED until the first live call. **Revisit when:** worker CPU contention appears (renders serialize with transcription on one loop) тАФ then either a dedicated render worker or flipping the env key.

**Captions server-side:** @napi-rs/canvas (Skia+HarfBuzz) rasterizes caption PNGs with vendored IBM Plex Sans Arabic TTFs (OFL) тАФ Arabic shaping visually verified before adoption (connected RTL letterforms). Same PNG-sequence overlay model; ffmpeg still never renders text.

**Measured (this desktop, 9.6-min stress clip, 13 segments, 137 captions):** render тЙИ 55s wall (encode sum 35.9s + download/captions/join) vs 1,030s in browser wasm тАФ ~15├Ч faster and the whole "don't close the tab" class of UX is gone. Railway shared-vCPU timing TBD on first deploy.

Append-only log of non-trivial decisions. Each entry: context тЖТ decision тЖТ why тЖТ revisit-when.

## 2026-07-08 тАФ Monorepo via npm workspaces
`apps/web` (Next.js/Vercel), `apps/worker` (Railway), `packages/core` (shared domain types: EDL, job contracts, caption tokens, filler lexicons, tier limits). Plain npm workspaces тАФ no Turborepo/Nx; a solo-scale build doesn't need build orchestration. **Revisit when:** package count or build times grow.

## 2026-07-08 тАФ Job queue: Postgres (Supabase) with FOR UPDATE SKIP LOCKED
No Redis/SQS/external queue. A `jobs` table + `claim_next_job()/complete_job()/fail_job()` SQL functions; the Railway worker polls every 2s over a direct Postgres connection. **Why:** zero extra infra/cost, transactional with domain data (enqueue + row update in one transaction), easily queryable status for the frontend (owners have read-only RLS on `jobs`). Retries: exponential backoff (30s┬╖2^attempts), max 3 attempts, `dedupe_key` for idempotent enqueues. **Revisit when:** sustained > ~10 jobs/sec or need for fan-out/priority lanes.

## 2026-07-08 тАФ Enum-like columns are text + CHECK constraints, not Postgres enums
Adding a value to a CHECK is a cheap constraint swap; altering enums is more ceremony (and was historically hazardous in migrations). **Revisit:** unlikely.

## 2026-07-08 тАФ Timeline direction: LTR always, even in the RTL UI
Media scrubbers/timelines flow leftтЖТright universally (players, NLEs, YouTube in Arabic). Mirroring the timeline would fight every user's muscle memory. So: the *chrome* (panels, labels, transcript) is RTL-first, but the timeline/scrubber component is hard-pinned LTR (`dir="ltr"` on the component, time 0 at the left). Play/pause/seek icons are NOT mirrored; back/forward navigation arrows in the chrome ARE logical. This is binding for Phase 3.

## 2026-07-08 тАФ i18n: next-intl, Arabic default at root path
`ar` is the default locale served at `/` (no prefix); English lives under `/en` (`localePrefix: "as-needed"`). Arabic is the design target; English adapts. All strings in `apps/web/messages/{ar,en}.json` тАФ no hardcoded UI strings anywhere. RTL via `dir` on `<html>` + Tailwind logical utilities (`ms-*`, `text-start`, тАж), never `ml-*`/`text-left` for directional layout.

## 2026-07-08 тАФ Browser locale detection disabled
next-intl's Accept-Language detection would send every English-configured browser to `/en` тАФ but Arabic creators very commonly run English OS/browsers. Since this is an Arabic-first product, `/` always serves Arabic; English is an explicit user choice at `/en` (`localeDetection: false`). **Revisit when:** signup data shows meaningful English-primary usage.

## 2026-07-08 тАФ Typography: IBM Plex Sans Arabic (single family, Arabic + Latin)
Loaded via `next/font`, covers both scripts so mixed-language lines don't change font mid-sentence. Also the default caption font in `@merai/core` caption specs.

## 2026-07-08 тАФ Retention windows (margin decision)
Raw footage: **30 days** after upload (`video_uploads.expires_at`), then storage object deleted by the `cleanup_expired` worker sweep (Phase 6). Final exports: **90 days**. Rationale: raw files dominate storage cost at ~90% target margin; 30 days comfortably covers an edit cycle. The EDL + transcript survive expiry, so the *edit* is never lost тАФ only re-rendering from raw becomes impossible. Windows are single constants in `@merai/core` (`RAW_RETENTION_DAYS`, `EXPORT_RETENTION_DAYS`) and migration defaults. **Revisit when:** users ask to re-edit older projects (offer paid retention as a Pro perk).

## 2026-07-08 тАФ Worker runs TypeScript directly via tsx (no build step)
`@merai/core` ships as TS source; web compiles it with `transpilePackages`, the worker executes it with tsx at runtime. Avoids a package build/watch pipeline for a solo team. **Revisit when:** worker startup time or memory matters.

## 2026-07-08 тАФ EDL is versioned, immutable, append-only JSON (v1 single-track)
`edl_versions` has insert+select RLS only тАФ no updates; every edit appends a version (cheap undo/audit). v1 locks audio+video together; Phase 5 J/L-cuts will introduce a v2 schema with decoupled tracks. The `version` literal in the JSON discriminates. Zod-validated on both producer and consumer sides (`@merai/core`).

## 2026-07-08 тАФ Transcripts: normalized words + raw provider payload both stored
`transcripts.words` is a provider-agnostic word array (stable ids, ms timestamps) that all downstream code uses; `transcripts.raw` keeps the full AssemblyAI response so Phase 2 analysis can be re-run or improved **without re-billing STT** (cost-relevant: STT is the largest per-unit cost).

## 2026-07-08 тАФ Auth: Supabase email+password to start
Email confirmation flow wired through a locale-less `/auth/confirm` route (handles both `token_hash` and PKCE `code`). OAuth (Google/Apple) deferred until there's user pull. Profile rows are created by a DB trigger on `auth.users` insert, carrying the signup locale.

## 2026-07-08 тАФ Next.js 16 conventions
Scaffolded on Next 16 (App Router, Turbopack default): `src/proxy.ts` instead of `middleware.ts` (nodejs runtime), async `params`/`cookies` everywhere. The proxy chains next-intl locale routing then Supabase session refresh on one response.

## 2026-07-08 тАФ Transcription behind a provider interface; mock is the keyless default
`TranscriptionProvider` (worker) with two implementations: `AssemblyAIProvider` (fully wired: request building, polling, error handling) and `MockTranscriptionProvider` (AssemblyAI-shaped Arabic/English fixtures with realistic word timings, fillers, and re-take gaps). The mock routes fixture data through the SAME normalization code as the real provider, so mock-based tests exercise the production path. Selection is env-only: a present `ASSEMBLYAI_API_KEY` activates the real provider; `TRANSCRIPTION_PROVIDER=mock|assemblyai` overrides. The key is read in exactly one place (`apps/worker/src/transcription/index.ts`). AssemblyAI runs with `disfluencies: true` so hesitation sounds survive into the transcript тАФ Phase 2's filler removal depends on them.

## 2026-07-08 тАФ 10-minute cap enforced in three layers, no ffprobe infra
(1) Browser probes duration from video metadata before any byte uploads (UX), (2) server action + DB CHECK validate the client-reported duration, (3) the transcription provider's measured `audio_duration` is authoritative тАФ over-long media permanently fails the upload/project and is never billed to usage. Running ffprobe server-side was deliberately skipped: it would require media tooling on Vercel/Railway for a check the STT provider gives us for free. **Trade-off:** a spoofed client duration is caught only after STT spend (тЙд10 min by the cap itself); acceptable at MVP scale.

## 2026-07-08 тАФ Pipeline status lives on projects.status; frontend polls
The job queue is an implementation detail; the UI-facing state machine is `projects.status` (uploading тЖТ transcribing тЖТ analyzing тЖТ ready | error), updated by server actions and the worker. Phase 1 goes transcribing тЖТ ready directly; Phase 2 inserts the analyzing step. The project page polls every 2.5s until terminal тАФ Supabase Realtime was deliberately skipped (publication setup + reconnect handling for zero MVP benefit). **Revisit when:** polling volume matters or live collaboration lands. Retries exhausted тЗТ worker marks the project error; the UI offers a retry action that requeues the job.

## 2026-07-08 тАФ DB-level tests run on PGlite applying the real migrations
No Docker on the dev machine, so tests use PGlite (Postgres-in-WASM, in-process): every worker test run applies the actual `supabase/migrations/*.sql` (validating the SQL itself) against stubbed `auth`/`storage` platform schemas, then drives the real claim тЖТ handler тЖТ complete/fail path. ## 2026-07-09 тАФ Export renders SEGMENT-WISE; single filter graphs are banned for cuts
The 10-minute stress test killed three single-command architectures in a row тАФ each ran out of memory **even in native ffmpeg** (32GB machine, 20GB free): (1) one overlay filter per caption line (139 concurrent overlays тЙИ 139 full-frame RGBA decoders), (2) N-branch trim+concat cuts (later segments' frames buffer in concat's input queues while earlier ones drain тАФ worst case approaches the whole decoded video in RAM), (3) a single-pass select/aselect graph. A plain full-length encode succeeded, isolating the filter graphs as the cause. **Architecture now:** one small ffmpeg run per kept segment (`-ss`/`-t` INPUT seeking тАФ only that window is ever demuxed/decoded), captions overlaid per segment as ONE concat-demuxer image sequence (PNGs + transparent gaps, windows clipped to segment-local time), then a final `-c copy` remux join. Peak memory is bounded by one segment regardless of total duration; frames are encoded exactly once (no generational quality loss); reordered timelines need no special path (segment file order = output order). Native validation on the 9.6-min clip: **36s total, join 0.3s**. Known trade-offs: AAC junctions can carry a ~20-40ms priming artifact at cut points (already discontinuities тАФ inaudible in practice), ~N├Ч ffmpeg startup + keyframe seek-decode overhead (negligible), and intermediate segment files live in the wasm FS until the join (source + PNGs deleted before joining to lower the ceiling). **Revisit when:** 1080p export or client CPUs make wasm encode times unacceptable тАФ then the fork is multithreaded core (COOP/COEP story) vs. server-side rendering.
